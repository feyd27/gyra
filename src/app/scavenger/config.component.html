What is Gyra?
Gyra is a tool that allows you to create formally valid and logically consistent configuration files for the Proof of Capacity miner Scavenger.
The configuration file used by Scavenger and created by Gyra is a .yaml file, which you can download in your browser and save it in the root directory where the Scavenger executable is placed.
get_app Download the Scavenger from the PoCC Github.

Security
Gyra is a front-end tool, with no back-end. All of the data inserted into the configuration form is processed only by built-in form validators. None of it is stored or sent outside the form. Gyra makes no HTTP requests whatsoever.
security Your mining account information is absolutely safe.

Features
  • Setups to go

    • star  Quick Setup: provide only the bare minimum - pool/wallet settings and plot directory paths, and you're good to mine. The rest of the configuration is populated by recommended default mining settings with CPU only.

    star  CPU-only Setup: configure only your pool/wallet, plot directories and CPU settings, and keep other parameters at recommended default values.

    star  GPU-only Setup: provide your pool/wallet information, paths to plot directories and configure the GPU you wish to mine with. All other configuration parameters are kept at default values, with the CPU switched off.

    star  Expert Setup: go through all configuration parameters and set each to your liking and hardware mining setup capabilities.
  • Instructions and documentation

    • star  The top toolbar: use the buttons to reset the configuration, preview and export it. Note that the buttons for configuration preview and export will be disabled as long as the provided settings contain parameters that are invalid or would create a logically inconsistent mining configuration.

    star  The bottom toolbar: set the level of application feedback you need and read detailed information about specific Scavenger parameters.

    error  Validation messages: each time a provided parameter formally doesn't comply with the configuration (e.g. a letter instead of a number), a validation message will be shown and the entry disabled. Once the error is corrected, the validation message will be cleared.

    info  Info messages: will be shown to indicate the estimated resource consumption - such as RAM usage with some settings.

    warning  Warning messages: will be shown each time the parameters are formally correct but make no sense (e.g. switching off both CPU and GPU in a configuration).

    help  Documentation: shows detailed information about specific parameters and their relevance in the context of the Scavenger Proof of Capacity miner.
feedback Share your feedback and improvements suggestions in the FeydLabs discord.

The quick setup requires the minimum information to create a valid mining setup. Just provide the pool address and plot file directories and you're good to mine. Any additional setting can also be applied, or just keep the defaults.
For the CPU-only setup, configure only your pool/wallet, plot directories and CPU settings, and keep other parameters at recommended default values. You can, however, modify any other configuration parameter.
For the GPU-only setup, configure only your pool/wallet, plot directories and GPU settings, and keep other parameters at recommended default values. You can, however, modify any other configuration parameter. Note that the CPU mining will be turned off once you clicked the GPU-only setup. Before completing the configuration, you can switch it back on by adjusting CPU settings.
Configure each parameter one by one. Note that each segment is collapsible - if you're not seeing a configuration group, click the button on the left side to show it.

  • HDD reader thread count: Sets the number of disks to be read in parallel, and it should reflect the number of hard disks where plot files are stored. In case the setting is left at default value of 0, Scavenger will use as many threads as there are disks.

    • Default value: no.

  • GPU threads: Number of GPU threads to use. Lower values show better performance. For integrated GPUs try the setting 1, for dedicated GPUs 1-4.
    • Default value: 0.
  • GPU worker task count: Refers to the number of threads used for GPU processing. In case no GPU is available on the system or it shouldn't be used, users should leave the default setting of 0.
    • Default value: 0.
  • GPU platform: GPU platform information.
    To list GPUs available on the system, users can run Scavenger with CLI option -o, and the Scavenger will display available GPU platforms and devices. It will also hint the ideal multiplier for the GPU nonce per cache setting.
    • Default value: 0.
  • GPU device: GPU device information. The way to find out the GPU device is described above.
    • Default value: 0.
  • GPU nonces per cache: GPU nonces per cache setting can improve the miner performance on some systems. It is practically impossible to give a general recommendation about this setting, as the performance depends on CPU settings and the overall system capability. Users can change this setting and observe how it will affect the performance, and adapt both this and the CPU nonces per cache setting until they reach the desired performance. To see the estimated RAM usage, click outside the form field after the parameter has been provided. Note that the GPU async setting also affects the RAM usage.
    • Default value: 262144.
  • GPU memory mapping: Enabling memory mapping can result in ZeroCopyBuffers for CPU, in setups with integrated GPUs, and it may speed up the data transfer on certain GPUs. The feature should be disabled for dedicated GPUs.
    • Default value: no.
  • GPU async: When set to "yes", it enables asynchronous data transfer and hashing, which may improve the performance of dedicated GPUs. The feature should be disabled for integrated GPUs.
    • Default value: no.

  • Console log settings: Defines the level of details shown in the console once Scavenger is running.
    • Default value: Info
  • Logfile settings Defines the level of details written in logfiles.
    • Default value: Warn
      Options available for logging:
    • Off No information will be displayed/logged. User are strongly advised against using this setting.
    • Error Information will be shown in the console or log will only when an error occurs.
    • Warn Warning and error information will be displayed/logged.
    • Info Shows/writes information on regular processes, including warnings and errors.
    • Debug Includes errors, warnings, regular information on processes and additional information relevant for debugging. Note that this setting will increase significantly the size of log files, and provides information of very little use to users with no development related skills
    • Trace Includes all above listed information with additional information regarding application process execution. Same as the "Debug" level - it significantly increases the size of log files and includes information relevant for development and optimization.
      • The general recommendation for the user who wishes to mine, without getting into the application structure and its working cycles, the "Info" level should provide enough information to monitor the application and the mining process. In case of repeated errors, unrelated to configuration and pool/wallet accessibility errors, the "Debug" setting can be turned on, to provide enough information to supporting developers.
  • Number of log files to keep: Sets the number of logfiles that will be available on the system.
    • Default value: 10
  • Log file size [MiB]: Sets the maximum size of a log file. Once the log file reaches the configured size, a new file will be written.
    • Default value: 20
  • Console log pattern: The formatting used to write information to the console.
    • Default value: Low noise
  • Logfile log pattern: The formatting used to write information to logfiles.
    • Default value: Low noise

Display and benchmark settings

error {{ formErrors.show_progress }}

error {{ formErrors.show_drive_stats }}

error {{ formErrors.benchmark_only }}

  • Show progress: When enabled, shows the progress and the speed of processing plot files.
    • Default value: Yes
  • Show drive statistics: When enabled, shows the statistics of processed hard drives.
    • Default value: No
  • Benchmark: If enabled, will run Scavenger in benchmark mode - without sending deadlines.The I/O option is used to test the maximum HDD reading speed. The XPU option is used for benchmarking the hashing speed.
    • Default value: Disabled
      For optimal performance in cases where both CPU and GPU mining is enabled, users are strongly advised to first run a benchmark and set the productive configuration based on results of it. One of the ways to complete the benchmark is:
    • optimizing CPU by setting the hdd_reader parameter to the number of hard drives with plot files,
    • proceed by setting the cpu_worker_thread_count value between the number of cores and hdd_reader and observe the performance depending on the settings,
    • optimize GPU (in GPU only scenario) by setting the value of gpu_worker_thread_count to values between 1 and hdd_reader,
    • experiment with gpu_nonces_per_cache and cpu_nonces_per_cache depending on available resources,
    • combine the CPU and GPU results in order to find the configuration for optimal performance.
